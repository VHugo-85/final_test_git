DESCRIPCIÓN GENERAL
-------------------
Construirás un pipeline de análisis de datos cardiovasculares
trabajando con Git desde el primer archivo. El análisis usa
el Heart Disease Dataset de la UCI, que se descarga
automáticamente con el notebook que recibes.

El foco no está en el análisis en sí, sino en cómo lo
construyes: cada sección tiene su propio commit, el historial
debe poder leerse como una historia y las ramas deben tener
un propósito claro.

ARCHIVOS QUE RECIBES
--------------------
  main_ayuda.ipynb  — el notebook con todo el código del proyecto
  guia.txt    — este archivo con todas las instrucciones

No necesitas crear ningún archivo de código.
Solo sigue las instrucciones en el orden en que aparecen.

HERRAMIENTAS NECESARIAS
-----------------------
  - Git instalado y configurado con tu nombre y correo
  - Python 3.9 o superior
  - Jupyter Notebook o JupyterLab (o VS Code con extensión Jupyter)
  - Una terminal (cmd, PowerShell, bash o la terminal de VS Code)

Antes de empezar, verifica que Git está configurado:

    git config --global user.name "Tu Nombre Completo"
    git config --global user.email "tu@correo.com"

Si ya lo configuraste antes, no necesitas hacerlo de nuevo.
Para verificarlo:

    git config --global user.name
    git config --global user.email

FASE 1 — Inicialización y commits por sección

El objetivo de esta fase es construir el pipeline de análisis
paso a paso, haciendo un commit después de cada sección del
notebook. Al terminar, git log --oneline debe mostrar una
historia coherente del proyecto.

----------------------------------------------------------------
PASO 1 — Crear la carpeta del proyecto
----------------------------------------------------------------

Crea una carpeta nueva con el nombre que quieras para el proyecto.
Puedes hacerlo desde el explorador de archivos o desde la terminal:

    mkdir proyecto_cardio
    cd proyecto_cardio

Desde ahora trabaja siempre dentro de esa carpeta.

----------------------------------------------------------------
PASO 2 — Inicializar el repositorio Git
----------------------------------------------------------------

Dentro de la carpeta del proyecto, ejecuta:

    git init

Git responderá algo como:
    "Initialized empty Git repository in .../proyecto_cardio/.git/"

Eso significa que el repositorio está creado. Todavía no hay
ningún commit, ningún archivo y ningún historial.

----------------------------------------------------------------
PASO 3 — Crear la estructura de carpetas
----------------------------------------------------------------

Crea dos carpetas vacías dentro del proyecto:

    mkdir data
    mkdir outputs

Git no versiona carpetas vacías. Para que Git las reconozca
necesitas crear un archivo placeholder dentro de cada una.
Ese archivo se llama por convención .gitkeep y está vacío.

En la terminal:

    Windows (cmd):
        type nul > data\.gitkeep
        type nul > outputs\.gitkeep

    Linux / macOS / Git Bash:
        touch data/.gitkeep
        touch outputs/.gitkeep

Tu estructura de carpetas debe verse así:

    proyecto_cardio/
        data/
            .gitkeep
        outputs/
            .gitkeep

----------------------------------------------------------------
PASO 4 — Crear el archivo .gitignore
----------------------------------------------------------------

El .gitignore le dice a Git qué archivos ignorar y no versionar.
En la raíz del proyecto (no dentro de data/ ni outputs/),
crea un archivo de texto llamado exactamente:

    .gitignore

Ábrelo con cualquier editor de texto y escribe dentro
el siguiente contenido tal como aparece aquí:

-------- CONTENIDO DE .gitignore --------
__pycache__/
*.pyc
*.pyo
.env
.DS_Store
data/*.csv
outputs/*.png
outputs/*.csv
-----------------------------------------

Guarda el archivo.

Este archivo le dice a Git que ignore los archivos CSV
que genera el notebook (los datos descargados), las imágenes
PNG que genera el EDA y los archivos de caché de Python.
Solo el notebook y los archivos de configuración se versionan.

FASE 2 — Corrección del historial
================================================================

El objetivo es practicar git reset para deshacer un commit
sin perder los cambios. El error es intencional y forma
parte del ejercicio.

----------------------------------------------------------------
PASO 1 — Ejecutar la celda de la Fase 2 en el notebook
----------------------------------------------------------------

En el notebook, busca la sección titulada:
    "FASE 2 — Corrección del historial"
Haz una copia a tu propio main.ipynb.

Ejecuta esa celda. Lo que hace internamente es leer la
proporción de enfermos que encontraste en el EDA y agregar
esa información al README.md automáticamente.

Cuando termine, abre el README.md y verifica que al final
del archivo apareció un bloque nuevo con el resultado.

----------------------------------------------------------------
PASO 2 — Hacer el commit con el mensaje malo (a propósito)
----------------------------------------------------------------

Haz staging de todos los cambios:

    git add .

Ahora haz el commit con este mensaje terrible, tal como aparece:

    git commit -m "cambios"

Verifica que quedó registrado:

    git log --oneline

Vas a ver el commit "cambios" al tope del historial.
Ese es el error que vas a corregir ahora.

----------------------------------------------------------------
PASO 3 — Deshacer el commit sin perder los cambios
----------------------------------------------------------------

Ejecuta:

    git reset HEAD~1

Este comando deshace el último commit pero conserva los cambios
en los archivos. Es como decirle a Git: "olvida que ese commit
existió, pero no toques mis archivos".

Verifica que el commit desapareció:

    git log --oneline

El commit "cambios" ya no debe estar en el historial.

Ahora verifica que los cambios siguen intactos:

    git status

Debes ver los archivos modificados en rojo bajo
"Changes not staged for commit". Los archivos están ahí,
simplemente ya no tienen commit asociado.

----------------------------------------------------------------
PASO 4 — Rehacerlo con un mensaje correcto
----------------------------------------------------------------

Haz staging de los cambios de nuevo:

    git add .

Ahora el commit con el mensaje correcto:

    git commit -m "docs: agregar resultado clave del EDA en README"

Verifica el historial:

    git log --oneline

El historial debe verse limpio, como si el commit "cambios"
nunca hubiera existido.

----------------------------------------------------------------
PASO 5 — Documentar la decisión en el README.md
----------------------------------------------------------------

Abre el README.md y agrega al final una sección nueva
con el siguiente contenido. Ajusta la oración en cursiva
con tus propias palabras:

-------- SECCIÓN A AGREGAR EN README.md --------
## Decisiones del historial

En la Fase 2 se usó `git reset HEAD~1` para deshacer un commit
cuyo mensaje no era descriptivo ("cambios"). Los archivos
modificados se conservaron y se volvieron a commitear con
un mensaje que explica claramente qué cambió y por qué.

Esta práctica es útil antes de hacer push al remoto, cuando
aún es posible reescribir el historial local sin afectar
a otros colaboradores.
-------------------------------------------------

Guarda el archivo y haz el commit:

    git add README.md
    git commit -m "docs: documentar correccion de historial en README"

================================================================
FUSIÓN DE LAS DOS RAMAS Y RESOLUCIÓN DEL CONFLICTO
================================================================

----------------------------------------------------------------
PASO 1 — Fusionar la primera rama (sin conflicto)
----------------------------------------------------------------

Estando en main, fusiona la primera rama:

    git merge feature/visualizaciones

Git dirá algo como:
    Updating xxxxxxx..xxxxxxx
    Fast-forward (o Merge made by the 'ort' strategy)

No hay conflicto aquí porque main no tiene cambios propios
que choquen con esa rama.

Verifica el historial:

    git log --oneline

Los commits de feature/visualizaciones ahora están en main.

----------------------------------------------------------------
PASO 2 — Fusionar la segunda rama (aquí viene el conflicto)
----------------------------------------------------------------

    git merge feature/estadisticas

Git se detendrá con un mensaje similar a este:

    Auto-merging README.md
    CONFLICT (content): Merge conflict in README.md
    Automatic merge failed; fix conflicts and then commit the result.

Esto es exactamente lo que esperábamos. Git encontró que ambas
ramas modificaron la misma parte del README.md y no puede
decidir solo cuál versión conservar.

----------------------------------------------------------------
PASO 3 — Entender el conflicto en el archivo
----------------------------------------------------------------

Abre el README.md con tu editor de texto.
Al final del archivo verás algo como esto:

-------- ASÍ SE VE EL CONFLICTO EN EL ARCHIVO --------
<<<<<<< HEAD
- Rama visualizaciones: boxplot de colesterol agregado.
=======
- Rama estadisticas: resumen por grupo de edad agregado.
>>>>>>> feature/estadisticas
-------------------------------------------------------

Qué significa cada parte:
  <<<<<<< HEAD         → inicio del bloque en conflicto
  (líneas de HEAD)     → versión que está en main actualmente
  =======              → separador entre las dos versiones
  (líneas de la rama)  → versión que viene de feature/estadisticas
  >>>>>>> feature/...  → fin del bloque en conflicto

----------------------------------------------------------------
PASO 4 — Resolver el conflicto manualmente
----------------------------------------------------------------

Edita el README.md. Elimina las tres líneas con marcas:
    <<<<<<< HEAD
    =======
    >>>>>>> feature/estadisticas

Y deja AMBAS líneas de contenido, una debajo de la otra,
exactamente así:

-------- CÓMO DEBE QUEDAR EL README.md DESPUÉS --------
- Rama visualizaciones: boxplot de colesterol agregado.
- Rama estadisticas: resumen por grupo de edad agregado.
-------------------------------------------------------

Guarda el archivo.

----------------------------------------------------------------
PASO 5 — Confirmar la resolución y cerrar el merge
----------------------------------------------------------------

Verifica que el conflicto está resuelto:

    git status

Debe aparecer README.md bajo "both modified" en rojo.
Agrégalo para indicar que lo resolviste:

    git add README.md

Cierra el merge con el commit:

    git commit -m "merge: integrar visualizaciones y estadisticas en main"

PASO 1 — Crear el repositorio en GitHub
----------------------------------------------------------------

1. Entra a https://github.com y accede a tu cuenta.

2. Haz clic en el botón "+" en la esquina superior derecha
   y elige "New repository".

3. Llena el formulario con estos valores:
       Repository name: proyecto_cardio   (o el nombre que prefieras)
       Description:     Pipeline de análisis cardiovascular — Curso Git
       Visibility:      Public

4. MUY IMPORTANTE: NO marques ninguna de estas opciones:
       - Add a README file        (ya tienes uno)
       - Add .gitignore           (ya tienes uno)
       - Choose a license

   Si marcas alguna, GitHub creará un commit propio y habrá
   conflicto al intentar conectar tu repositorio local.

5. Haz clic en "Create repository".

GitHub te mostrará una página con instrucciones.
Necesitarás la URL del repositorio que aparece arriba.

       https://github.com/tu_usuario/proyecto_cardio.git

PASO 2 — Generar un token de acceso personal (si usas HTTPS)
----------------------------------------------------------------

GitHub ya no acepta contraseñas para hacer push.
Necesitas un token personal. Si ya tienes uno configurado
puedes saltar al Paso 3.

Para crear uno:

1. En GitHub, ve a tu foto de perfil (esquina superior derecha)
   → Settings → Developer settings → Personal access tokens
   → Tokens (classic) → Generate new token (classic).

2. En "Note" escribe: curso-git-proyecto-final

3. En "Expiration" elige 30 days (suficiente para el proyecto).

4. En "Select scopes" marca:  repo  (marca la casilla principal,
   selecciona todos los sub-permisos automáticamente).

5. Haz clic en "Generate token".

6. COPIA EL TOKEN AHORA. GitHub no te lo mostrará de nuevo.
   Guárdalo en un lugar seguro (no lo pongas en el código).

   Tiene esta forma:
       ghp_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

PASO 4 — Subir la rama main al remoto
----------------------------------------------------------------

La primera vez que hagas push, la terminal pedirá
tus credenciales de GitHub:

    Password: (aquí pega el token, no tu contraseña)

El token no se muestra mientras lo escribes, eso es normal.
Presiona Enter.

Si el push fue exitoso, verás algo como:
    Branch 'main' set up to track remote branch 'main' from 'origin'.
    * [new branch] main -> main

Pregunta 1
